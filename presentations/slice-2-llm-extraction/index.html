The LLM's job is narrow: read a passage and return a small, typed object — sentiment, strengths, concerns, domain depth. It never sees a name, and it never sets a rank.

Two new stages slot in before matching; everything downstream is the deterministic core from Slice 1, unchanged.

The new data flow3 / 8

Guardrail #1 — the release blockerNothing leaves the machine un-scrubbed

This is the most important test in the whole project. Before any text reaches the hosted LLM, a scrubber replaces personal data with tokens — and a test captures every outbound payload to prove it.

Scrubbed: names, emails, phone numbers, client names. Allowed through: skills, anonymised experience, feedback content. If the test ever sees raw PII leaving, the build is blocked.

The PII gate4 / 8

Guardrail #2 — no invented praiseEvery claim must cite the source text

The LLM must return the exact passage that supports each claim. If that passage isn't actually in the source, the claim is dropped — not scored.

This stops a confident model from inventing a strength nobody wrote. A dropped claim flags the candidate, and the dimension falls back to the neutral 50 rather than trusting a hallucination.

What lights up6 / 8

Keeping it cheap and stableThe LLM is metered, not let loose

Cached

Each passage is extracted once and cached. Re-runs cost nothing — a test asserts zero new calls on a warm cache.

Temperature 0

Deterministic decoding. The same text yields the same signal, so bands stay stable run-to-run.

Bounded

A fixed, counted number of calls per consultant — a cost test fails if it drifts upward.

Reproducibility shifts from "byte-identical" (Slice 1) to "stable bands & signals" — exact rank may shuffle by one, and that's acceptable for a human-review list.
