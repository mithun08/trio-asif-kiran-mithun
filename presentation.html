The Problem We're Solving

Today's reality
  • Staffing managers match consultants to roles by hand in a spreadsheet
  • Slow — decision latency slows bench-to-role placement
  • Inconsistent — judgment varies person-to-person, meeting-to-meeting
  • No audit trail — hard to justify or revisit recommendations
  • Growing fast — ~5 % headcount per month; spreadsheets don't scale
Supply pool: Beach · Rolling-off · New joiners — all in constant flux
What we want
  • A ranked, explainable shortlist for any open role
  • Surfaces trade-offs so a human makes the final call
  • Consistent, reproducible scoring across runs
  • Privacy-safe — PII scrubbed before any cloud call
  • Works on a laptop — no server infra required in v1

Design Philosophy

Principle 1
Deterministic
Core

Filtering, weighting, ranking, and tiebreaks are pure Python — no randomness, no cloud call. The LLM is used only at the edges: extraction and explanation.

Principle 2
LLM Never
Sets Rank

LLM explains a rank already computed deterministically. A hallucinated explanation cannot change who is recommended.

Principle 3
Privacy
First

PII is scrubbed via Presidio before any text leaves the machine. Embeddings are generated locally — no profile text sent off-device for similarity.

Principle 4
Cost
Discipline

Slice-1 = zero LLM calls. LLM is added only where unstructured input genuinely needs it. DSPy response caching makes warm runs free.

Principle 5
Config
Driven

All weights, thresholds, and model IDs live in config/default.yaml. Zero magic numbers in source code.

Principle 6
Testable
Stages

Each pipeline stage is independently unit-testable with no cross-stage coupling. Evals gate both LLM extractions and explanations.

LLM Extraction — How It Works

What the LLM extracts
DimensionSignals extracted
Feedback Quality Sentiment (positive/neutral/negative), client-keep signal, domain depth, concerns
Adaptability Tech transitions ≥ 2, learning-speed mentions, cross-domain ≥ 2, upskilling evidence
Performance Trend improving / stable / declining / unknown
Grounding invariant: each extracted claim must be traceable to a span in the source text. Ungrounded extractions are flagged, not scored.
DSPy modules (typed signatures)
  • ProfileExtraction — free-form PDF → structured fields
  • FeedbackSignalExtraction — markdown feedback → signals
  • AdaptabilitySignalExtraction — combined text → adaptability flags
  • PerformanceTrendExtraction — text → trend direction
  • CandidateExplanation — evidence package → NL explanation
Model tier: gpt-4o-mini for extraction (high-volume, cheap) · gpt-4o for explanations (low-volume, quality matters) · claude-3-haiku fallback

Privacy & Security

PII pipeline (Presidio + spaCy)
  • Scrub-before-send: names, emails, phone numbers, client names removed/pseudonymised before any API call
  • Token map stored locally — rehydrated after LLM returns, never in transit
  • PII gate test is a release blocker
  • Entities detected: PERSON, EMAIL_ADDRESS, PHONE_NUMBER, ORGANIZATION
Embeddings generated locally with all-MiniLM-L6-v2 — no profile text leaves the machine for the similarity step.

Why This Approach Works

  • Explainable by design — every score carries evidence strings traceable to source data; no black-box ranking
  • Privacy-safe — PII gate is a release blocker; embeddings never leave the machine
  • Cost-efficient — ~$0.02 cold, $0.00 warm; embeddings are free and cached
  • Reproducible — deterministic core + DSPy caching makes re-runs byte-identical on warm cache
  • Resilient to missing data — degraded confidence, not dropped candidates
  • Independently testable stages — each pipeline step has its own unit-test surface; no integration-only coverage
  • Config-driven flexibility — weights and thresholds tunable without code changes
  • Scales gracefully — Milvus Lite handles POC; swap to managed Milvus when volume demands
  • Two modes — --no-llm for fast offline iteration; LLM-on for full-signal production reports
  • Audit trail — structured run log: timings, token spend, data-quality flags per run
Human stays in the loop: the tool surfaces trade-offs and surfaces gap candidates when no one passes hard filters — it never makes the placement decision.
