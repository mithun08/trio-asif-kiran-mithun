📥 Ingest Stage  — Deterministic

Functions

  • ingest_roles(xlsx_path)
    Parses "Open Roles" sheet from Excel workbook
    → list[Role]
  • ingest_consultants_from_workbook(xlsx_path)
    Reads 3 supply-state sheets: Beach, Rolling Off, New Joiners
    → list[Consultant] (no skills yet)
  • ingest_consultants(profiles_dir, workbook_consultants)
    Enriches with raw PDF text via Docling + optional OCR
    → list[Consultant] with raw_profile_text
  • ingest_feedback(feedback_dir, consultants)
    Parses markdown files — 3 sections per consultant: project / client / beach
    → list[Consultant] enriched with feedback_text

Why No LLM?

Reason

Ingest is purely data collection. Excel rows map 1-to-1 to typed fields; PDFs are read as raw text only (semantic extraction happens later in Extract). Using an LLM here would incur cost and latency with zero benefit — the data is already structured.

Docling handles PDF→text conversion deterministically. OCR is Tesseract-based, not neural. Data gaps (unreadable PDFs, OCR failures) are tracked as data_gaps flags on the Consultant model.

Data Gaps Tracked

  • profile_pdf_unreadable — PDF could not be parsed
  • profile_pdf_ocr_used — OCR fallback was triggered
  • profile_pdf_low_confidence — OCR confidence < 0.6
  • profile_pdf_unmatched — PDF exists but no workbook row found

🔧 Normalise Stage  — Deterministic

Functions

  • canonicalise_locations(consultants)
    Static map: "bangalore" → "Bengaluru", "remote india" → "Remote-India"
    → list[Consultant] with normalised locations
  • dedup_by_email(consultants)
    Removes duplicates keyed by email, keeps first occurrence
    → de-duplicated list[Consultant]
  • scrub_pii(consultants)
    Presidio + spaCy en_core_web_sm NER. Replaces emails, phones, names with tokens. Validates no residual PII via regex.
    → scrubbed_text + pii_token_map {<EMAIL_0>: "john@x.com"}

Why No LLM for PII Scrubbing?

Critical Reason

Sending PII-containing text to an external LLM before scrubbing it would be a data-privacy violation. Presidio runs fully locally using rule-based patterns + spaCy NER — no text leaves the machine.

Using an LLM to scrub PII would also be non-deterministic. Presidio is auditable: you get a structured entity list with offsets, which is turned into a reversible token map for later rehydration in Explain.

Token Map (Rehydration)

Scrubbed text passed to LLM contains: <PERSON_0>, <EMAIL_ADDRESS_1>
After LLM generates explanation, rehydrate_text() replaces tokens back with original values for human-readable output.

🤖 Extract Stage  — LLM (gpt-4o-mini via OpenRouter)

DSPy Signatures

  • ProfileExtraction
    Input: raw_profile_text (scrubbed PDF). Output: skills_json, location, grade, evidence_spans
    Runs once per consultant with a PDF
  • FeedbackSignalExtraction
    Input: feedback_text per source. Output: sentiment, strengths, concerns, client_keep_signal, domain_depth, evidence_spans
    Runs per consultant per feedback source (project / client / beach)
  • AdaptabilitySignalExtraction
    Input: combined profile + feedback. Output: tech_transitions (int), cross_domain (int), learning_speed_mentions, upskilling (booleans)
    Runs once per consultant
  • PerformanceTrendExtraction
    Input: combined text. Output: improving / stable / declining / unknown
    Runs once per consultant

Why LLM here?

Reason

PDF CVs and markdown feedback are unstructured prose. Rules cannot reliably extract "3 tech-era transitions" or "client requested retention" from narrative text. Only an LLM can interpret semantic meaning at this level.

All 4 signatures require evidence_spans — verbatim substrings grounding each claim. This prevents hallucinated signals from propagating into scoring.

Cost Controls

  • Incremental cache: results stored by source_hash. Re-extraction only on file change.
  • Model: openai/gpt-4o-mini (cheapest capable model)
  • Concurrency: max 5 parallel extractions
  • Budget guard: max_cost_usd_per_run check after each consultant
  • Prompt injection defense: all inputs wrapped with document boundary markers

🗄️ Index Stage  — Deterministic (local embedding)

Functions

  • build_index(consultants, roles, index_dir)
    Embeds each consultant skill name with all-MiniLM-L6-v2 (384-dim). Stores in Milvus Lite at .cache/milvus/
    Fields stored: consultant_email, skill_name, vector
  • load_index(index_dir)
    Loads existing Milvus collection at match time. Returns client + model handle.
    → (MilvusClient, SentenceTransformer)
Purpose in Scoring
When a required skill has no exact or adjacency match, the skill_match dimension queries Milvus: embed the required skill → search consultant's skill vectors → if cosine distance ≥ 0.65, grant c_vector = 65 credit.
Example: Role needs "FastAPI" → consultant has "Flask" → vector similarity 0.72 → credit 65 (between adjacent 60 and exact 100).

Why No LLM?

Reason

all-MiniLM-L6-v2 is a sentence-transformer — a small (22M param) encoder that runs fully locally. No text leaves the machine. It is deterministic, fast (<10ms per embedding), and sufficient for skill-name similarity.

Using an LLM to judge skill similarity would be 100× more expensive and non-deterministic. The vector index only supplements the skill adjacency map — it is not the primary matching mechanism.

Vector Similarity Threshold

Threshold: skill_vector_similarity = 0.65 (cosine distance)
Credit awarded: c_vector = 65 (between adjacent 60 and exact 100)
Set in config/default.yaml — not hardcoded

⚖️ Match Stage  — Fully Deterministic

Hard Filters (pre-scoring elimination)

  • Availability Filter
    Beach consultants: always pass. Rolling-off: must be within 5-day buffer of role start_date (10-day for medium confidence). New joiners: within 7-day buffer.
  • Location Filter
    Only applies if role.co_located = true. Consultant primary location must match role's first location.

6 Scoring Dimensions

  • skill_match (35%)
    exact(100) → proficiency-short(70) → adjacent(60) → vector-sim(65) → new-joiner(40)
  • feedback_quality (25%)
    project(50%) + client(30%) + beach(20%) weighted by sentiment + keyword bonuses
  • availability (15%)
    Linear decay over 30-day horizon; rolloff confidence penalty (0/10/30%)
  • adaptability (15%)
    Base 50 + bonuses: transitions(+15), cross-domain(+10), upskilling(+10), learning(+10)
  • supply_state (5%)
    beach(100) → rolling_off(70) → new_joiner(40)
  • performance_trend (5%)
    improving(100) → stable(70) → declining(30) → unknown(50)

Ranking & Bands

  • ranker.rank_candidates()
    Sort: total_score DESC → availability DESC → data_confidence DESC → supply_state DESC. Ties share rank.
  • Strong ≥ 75  ·  Partial ≥ 40  ·  Gap < 40

Why No LLM?

Core Design Principle

The LLM never sets a rank. Ranking must be deterministic and auditable. LLM rankings are non-deterministic, cannot be explained by pointing to specific config values, and cannot be re-run reproducibly.

All signal extraction was done in the Extract stage. Match only applies arithmetic formulas. Every score can be traced back to a config value in config/default.yaml and an evidence string in the dimension output.

Gap Analysis

If all candidates fail hard filters: re-run with filters disabled to show why each was rejected. Bench distribution (grade × supply_state) reported for capacity planning.

💬 Explain Stage  — LLM (gpt-4o via OpenRouter)

DSPy Signature

  • CandidateExplanation
    Inputs: role_title, candidate_name (rehydrated), dimension_scores_json (all 6), why_not_higher_context (gap vs rank above)
    Outputs: explanation (2-3 sentences), why_not_higher (1-2 sentences, empty for rank 1)
    Validation: explanation must reference dimension vocabulary

Post-Processing

  • PII tokens rehydrated: <PERSON_0> → "Alice Patel" before display
  • Fallback model: anthropic/claude-3-haiku if primary fails
  • Skip with --no-explanations flag to save cost

Why LLM here?

Reason

Natural language generation of trade-off reasoning cannot be done deterministically. "Alice scores high on skills but her rolling-off date adds risk" requires synthesising multiple dimension values into coherent prose — this is exactly what LLMs are for.

The LLM only narrates the scores — it does not recompute or change them. If the explanation contradicts the numbers, the numbers win.

Cost Controls

  • Only runs for top-N candidates (default 5). A 20-consultant bench → 5 LLM calls, not 20.
  • Model: gpt-4o — higher quality justified here since this is user-facing output

3LLM Decision Matrix — Why each stage is or isn't LLM-powered

Stage LLM? Model Why / Why Not
📥 Ingest ✕ No LLM — Data is already structured (Excel rows) or just raw text being collected (PDF text, markdown). No semantic interpretation needed yet. Docling handles PDF→text deterministically.
🔧 Normalise ✕ No LLM — PII scrubbing must run locally before any text leaves the machine — using an LLM for scrubbing would send unredacted PII to an external API. Presidio + spaCy provide auditable, deterministic PII detection. Location canonicalisation is a simple static map.
🤖 Extract ✓ LLM openai/gpt-4o-mini CV text and feedback are unstructured prose. Detecting "3 tech-era transitions", "client keep signal", or "improving trend" from narrative text requires semantic understanding that rules cannot provide. Results are cached by file hash — LLM cost is amortised across all future match runs.
🗄️ Index ✕ No LLM all-MiniLM-L6-v2 (local) Sentence-transformer embeddings are deterministic, fast, and run entirely locally (22M param model). No text leaves the machine. LLM-quality semantic judgement is not needed for skill-name similarity — a small encoder is sufficient and 100× cheaper.
⚖️ Match ✕ No LLM — Scoring must be reproducible and auditable. Every score traces back to a config value in default.yaml and an evidence string. LLM rankings cannot be re-run deterministically and cannot be explained by pointing to specific config values. The LLM never sets a rank.
💬 Explain ✓ LLM openai/gpt-4o Natural language synthesis of trade-off reasoning cannot be done deterministically. The LLM only narrates the pre-computed scores — it does not change them. Constrained to top-5 candidates to cap cost. PII tokens are rehydrated post-generation so the LLM never sees real names/emails.
